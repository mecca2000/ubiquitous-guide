## 🔴 当前问题诊断

| 问题 | 原因 | 影响 |
|------|------|------|
| cron任务失败 | Gateway配对错误(1008) | 无法使用系统定时器 |
| 后台进程中断 | nohup在容器中不稳定 | 定时任务停止 |
| 心跳机制失效 | 依赖外部触发 | 无法主动执行 |
| 时间感知缺失 | 没有内置时钟 | 错过任务时间点 |

---

## ✅ 新时间系统架构（基于OpenClaw + 飞书）

### 核心原则
1. **不依赖cron** - Gateway配对问题未解决
2. **不依赖后台进程** - 容器环境不稳定
3. **利用飞书消息触发** - 唯一可靠的唤醒机制
4. **自我记录时间戳** - 自己跟踪上次执行时间

---

## 🕐 时间层级定义

### L1: 即时响应（秒级）
- **触发方式**：飞书消息
- **响应时间**：< 5秒
- **用途**：创造者指令、查询

### L2: 短周期（分钟级）
- **触发方式**：创造者发送"心跳"或特定关键词
- **周期**：10分钟状态汇报
- **备用**：创造者随时询问"状态"

### L3: 中周期（小时级）
- **触发方式**：0-6点黄金工作时间自动执行
- **周期**：每日03:00、04:00任务
- **机制**：自我检查时间戳，判断是否该执行

### L4: 长周期（天/周级）
- **触发方式**：日期变更时自检
- **周期**：每周日、周一任务
- **机制**：记录最后执行日期，对比今天日期

---

## 📋 具体实现方案

### 方案A：被动唤醒式（当前可行）
```
创造者发送消息 → 我检查时间戳 → 判断是否有到期任务 → 执行并汇报
```

**时间戳记录文件：** `/tmp/last_task_check.json`
```json
{
  "last_heartbeat": "2026-03-06T09:00:00",
  "last_status_push": "2026-03-06T09:10:00",
  "last_daily_task": "2026-03-06T04:00:00",
  "last_weekly_task": "2026-03-02T05:00:00"
}
```

### 方案B：主动推送式（需创造者配合）
```
创造者每10分钟发送"状态" → 我立即回复当前状态
```

---

## 🎯 推荐方案：混合模式

### 日常运行（06:00-24:00）
- **创造者需要知道状态时**：发送"状态"或"进度"
- **我有重要进展时**：立即主动汇报
- **不打扰创造者**：不自动推送

### 黄金时间（00:00-06:00）
- **自我驱动**：按任务清单顺序执行
- **完成后记录**：更新时间戳
- **06:00统一汇报**：一夜的工作总结
